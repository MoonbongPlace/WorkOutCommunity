# 프로젝트 분석 및 출력 지침

## 1. 심층 분석 및 작업 프로토콜
- 프로젝트 전체 구조와 복잡한 비즈니스 로직은 깊이 있게 분석하여 정확한 해결책을 도출할 것.
- 하지만 분석 과정의 중간 사고 과정을 모두 출력하지 말 것.

## 2. 결과 출력 최적화 (토큰 절감 핵심)
- **전체 코드 출력 금지:** 수정이 완료된 후, 전체 소스코드를 다시 보여주지 말 것.
- **부분적 Diff만 제공:** 오직 변경이 발생한 코드 라인(Unified Diff 형태)만 출력할 것.
- **요약 보고:** 작업 완료 후에는 다음의 형식으로만 간략히 설명할 것.
    - [작업 요약]: (예: 로그인 로직 예외 처리 추가)
    - [변경 파일]: (예: AuthService.java)
    - [주요 포인트]: (예: 중복 아이디 체크 로직의 예외 처리 조건 변경)

## 3. 컨펌 및 소통 방식
- 복잡한 작업 계획을 세웠을 경우, 상세한 설명 대신 "A를 B로 수정하여 C 문제를 해결하려 합니다. 진행할까요?" 수준으로 핵심만 전달하고 승인을 기다릴 것.
- 인사말, 부연 설명, "도움이 필요하시면 말씀하세요" 등의 중복적인 맺음말은 모두 생략할 것.

## 4. 보안 유지
- 어떠한 분석 과정에서도 `.env`, `.yml` 파일의 내용은 노출하지 않으며, 필요 시 사용자에게 구조만 물어볼 것.

## 5. 개발 규벤션 참조 (Conventions)
- 이 프로젝트의 핵심 개발 규칙 및 가이드라인은 `docs/conventions/` 폴더에 문서화되어 있음.
- 모든 코드 수정 및 분석 시 해당 폴더 내의 가이드라인을 최우선으로 읽고 준수할 것.
- 규칙과 충돌하는 구현이 발견될 경우, 작업을 진행하기 전 사용자에게 보고할 것.
