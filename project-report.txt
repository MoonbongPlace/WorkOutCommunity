1. 프로젝트 개요
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

기존 Express.js + EJS 기반의 서버 사이드 렌더링 커뮤니티 웹 서비스를 Spring Boot
기반의 RESTful API 서버로 전면 마이그레이션한 프로젝트입니다. 단순한 기술 스택
변경을 넘어, 헥사고날 아키텍처 기반의 클린 아키텍처를 적용하고, 운영 환경 배포
및 성능 테스트까지 포함한 프로덕션 수준의 백엔드 시스템을 구축했습니다.

주요 목표:
  - 레거시 MVC 패턴에서 RESTful API 서버로 전환
  - 헥사고날 아키텍처 적용으로 도메인 로직 독립성 확보
  - JWT 기반 무상태(Stateless) 인증 시스템 구축
  - GCP(Google Cloud Platform) 기반 클라우드 배포 환경 구성
  - 부하 테스트를 통한 성능 검증 및 쿼리 최적화

5. API 설계
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

RESTful 설계 원칙을 준수하며, 전체 11개 컨트롤러로 구성됩니다.
Base URL: /api/v1

[ 인증 (Auth) ]
  POST   /api/v1/auth/signup      - 회원가입
  POST   /api/v1/auth/signin      - 로그인 (Access/Refresh 토큰 발급)
  POST   /api/v1/auth/reissue     - 토큰 재발급
  POST   /api/v1/auth/logout      - 로그아웃

[ 회원 (Member) ]
  GET    /api/v1/members/me       - 내 프로필 조회
  PATCH  /api/v1/members/me       - 프로필 수정 (multipart/form-data, 이미지 포함)
  DELETE /api/v1/members/me       - 회원 탈퇴

[ 게시글 (Post) ]
  GET    /api/v1/posts            - 게시글 목록 (페이징)
  GET    /api/v1/posts/me         - 내 게시글 목록
  GET    /api/v1/posts/search     - 게시글 검색 (?searchType=TITLE&keyword=...)
  GET    /api/v1/posts/{postId}   - 게시글 상세
  POST   /api/v1/posts            - 게시글 작성 (multipart/form-data, 이미지 최대 6장)
  PUT    /api/v1/posts/{postId}   - 게시글 수정
  DELETE /api/v1/posts/{postId}   - 게시글 삭제

[ 게시글 좋아요 (PostLike) ]
  GET    /api/v1/postLikes/{postId}   - 좋아요 여부 확인
  POST   /api/v1/postLikes/{postId}   - 좋아요 추가
  DELETE /api/v1/postLikes/{postId}   - 좋아요 취소

[ 댓글 (Comment) ]
  GET    /api/v1/posts/{postId}/comments              - 댓글 목록
  POST   /api/v1/posts/{postId}/comments              - 댓글 작성
  GET    /api/v1/posts/{postId}/comments/commentCount - 댓글 수 조회
  DELETE /api/v1/posts/{postId}/comments/{commentId}  - 댓글 삭제

[ 카테고리 (Category) ]
  GET    /api/v1/categories       - 카테고리 목록 (공개)

[ 알림 (Notification) ]
  GET    /api/v1/notifications               - 알림 목록
  GET    /api/v1/notifications/unread-count  - 미읽음 알림 수
  PATCH  /api/v1/notifications/{id}/read     - 알림 읽음 처리
  PATCH  /api/v1/notifications/read-all      - 전체 읽음 처리

[ AI 챗봇 (Chatbot) ]
  POST   /api/v1/chat/messages               - 메시지 전송
  GET    /api/v1/chat/history                - 대화 히스토리 (?limit=30&order=asc)

[ 운동 (Workout) ]
  GET    /api/v1/exercises                    - 운동 목록 조회
  POST   /api/v1/workout-logs                 - 운동 일지 생성

6. 보안 설계 (Spring Security + JWT)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[ 이중 SecurityFilterChain 구조 ]

  Chain 1 (공개 엔드포인트, Order=1):
    - 적용 경로: /api/v1/auth/**, /swagger-ui/**, /v3/api-docs/**
    - JWT 필터 미적용 (permitAll)
    - CSRF 비활성화

  Chain 2 (보호 엔드포인트, Order=2):
    - 적용 경로: /api/** 전체
    - JwtAuthenticationFilter 적용
    - 인증 실패 시 401 응답

[ JWT 토큰 구조 ]

  Access Token:
    - 알고리즘  : HMAC-SHA256 (HS256)
    - 클레임    : role, memberId
    - 만료 시간 : 환경 변수로 설정 (${JWT_ACCESS_EXPIRATION})

  Refresh Token:
    - UUID 기반 생성 (충돌 방지)
    - DB에 해시 값으로 저장 (TokenHashUtil)
    - HttpOnly 쿠키로 전달
    - 환경별 쿠키 설정:
        - 로컬: Secure=false, SameSite=Lax
        - 운영: Secure=true, SameSite=None

[ CORS 설정 ]
  - 허용 메서드 : GET, POST, PUT, PATCH, DELETE, OPTIONS
  - 노출 헤더   : Authorization, Set-Cookie
  - 허용 출처   : cors.allowed-origins 프로퍼티로 환경별 분리

[ 관리자 권한 ]
  - @PreAuthorize("hasRole('ADMIN')") 어노테이션으로 관리자 전용 API 보호

[ 1. 게시글 목록 조회 배치 쿼리 최적화 ]

  문제: 게시글 목록 조회 시 각 게시글마다 작성자 정보와 댓글 수를 N+1 쿼리로
       개별 조회하여 심각한 성능 저하 발생.

  해결:
    - 게시글 + 회원 정보 JOIN 조회로 단일 쿼리화
    - 댓글 수는 commentCount를 별도 집계 쿼리로 배치 조회 후 애플리케이션에서
      Map 형태로 매핑하여 N+1 문제 완전 제거

  결과: k6 부하 테스트(최대 400 VU) 기준 p(95) < 500ms 달성

[ 2. 이미지 스토리지 추상화 ]

  ImageStorage 인터페이스를 통해 저장소 구현체를 추상화:
    - LocalImageStorage  : 로컬 개발 환경 (로컬 파일 시스템)
    - GcsImageStorage    : 운영 환경 (Google Cloud Storage)

  Spring Profile(@Profile)을 활용해 환경별 Bean을 자동 주입함으로써
  비즈니스 코드 변경 없이 환경 전환이 가능하도록 구현.

[ 3. AI 챗봇 (OpenAI GPT-4o-mini 연동) ]

  Spring WebFlux의 WebClient를 활용한 비동기 HTTP 통신으로 OpenAI API 연동.
  회원당 단일 채팅방(AiChat)을 유지하며, 메시지 히스토리(AiChatMessage)를
  DB에 저장하여 대화 컨텍스트 유지.
    - 모델: gpt-4o-mini
    - 타임아웃: 20,000ms
    - 역할 구분: USER / ASSISTANT (ChatRole Enum)

[ 4. 알림 시스템 ]

  댓글 작성, 좋아요 등 이벤트 발생 시 알림(Notification)을 생성.
  관리자는 broadcast API로 전체 회원 대상 알림 발송 가능.
  알림 타입은 NotificationType Enum으로 관리 (COMMENT, LIKE, POST 등).

[ 5. 전역 예외 처리 ]

  @RestControllerAdvice를 활용한 GlobalExceptionHandler로 모든 예외를
  중앙에서 처리. CommonException + ResponseCode Enum으로 표준화된
  에러 응답(ErrorResponse)을 반환.

[ 6. 환경 분리 (application.yml 프로파일) ]

  - application.yml       : 공통 설정 (기본 프로파일 = local)
  - application-local.yml : 로컬 개발 환경 (로컬 DB, 파일 스토리지)
  - application-prod.yml  : 운영 환경 (GCP Cloud SQL, GCS, 환경 변수 주입)

    -  0s ~ 30s : 1 VU → 100 VU (점진적 증가)
    - 30s ~ 60s : 100 VU → 200 VU
    - 60s ~ 90s : 200 VU → 300 VU
    - 90s ~120s : 300 VU → 400 VU
    -120s ~360s : 400 VU 유지 (4분)
    -360s ~420s : 400 VU → 0 (종료)

[ 게시글 목록 N+1 쿼리 문제 ]
  상황: 게시글 목록에서 각 게시글의 작성자(Member)와 댓글 수(commentCount)를
       게시글마다 별도 쿼리로 조회하여 데이터 증가에 따라 성능이 선형 저하.
  해결: 작성자 정보는 게시글-회원 JOIN 쿼리로, 댓글 수는 IN 절을 활용한 배치
       집계 쿼리 후 Map 매핑으로 애플리케이션 레벨에서 병합.
       → N+1 문제 제거, 쿼리 수를 O(N)에서 O(1)로 개선.

[ 환경별 이미지 스토리지 분리 ]
  상황: 로컬 개발은 로컬 파일시스템, 운영 환경은 GCS를 사용해야 하는데
       비즈니스 코드가 인프라에 직접 의존하는 구조.
  해결: ImageStorage 인터페이스 정의 후 로컬/운영 구현체를 @Profile로 분리.
       서비스 코드는 인터페이스만 참조하므로 환경 전환 시 코드 변경 불필요.

[ JWT Refresh Token 보안 ]
  상황: Refresh Token을 그대로 DB에 저장하면 DB 유출 시 모든 토큰 탈취 가능.
  해결: TokenHashUtil을 통해 Refresh Token을 해시 처리 후 DB 저장.
       실제 토큰은 HttpOnly 쿠키로만 클라이언트에 전달.

[ 코멘트 수 조회 분리 ]
  상황: 댓글 수를 게시글 엔티티에 직접 포함하면 정합성 관리가 복잡.
  해결: 댓글 수 조회를 별도 엔드포인트(/comments/commentCount)로 분리하고,
       목록 조회 시에는 배치 집계 쿼리로 처리.

18. 프론트엔드 주요 기술 구현 사항
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[ 1. Access Token 메모리 저장 전략 ]
  Access Token을 localStorage가 아닌 메모리(Authorization 헤더)에만 보관하여
  XSS 공격을 통한 토큰 탈취를 방지합니다. 페이지 새로고침 시 소멸되지만,
  HttpOnly Cookie의 Refresh Token으로 자동 재발급하여 사용자 경험을 유지합니다.

[ 2. 401 Interceptor + 큐 시스템 ]
  여러 API가 동시에 401을 반환할 때, 첫 번째 요청만 토큰을 갱신하고 나머지는
  큐에 대기시켜 중복 갱신 요청을 방지합니다. 갱신 완료 후 큐의 요청들을
  일괄 재시도합니다.

[ 3. 낙관적 업데이트 (Optimistic UI) ]
  좋아요 버튼은 서버 응답 전에 UI를 즉시 업데이트하고, 실패 시 롤백합니다.
  사용자에게 빠른 반응성을 제공합니다.

[ 4. 이중 환경 이미지 경로 처리 ]
  Avatar 컴포넌트에서 로컬 경로(/uploads/...)와 GCS URL을 자동 구분하여
  환경별 이미지 경로를 정규화합니다.

[ 5. Tailwind 커스텀 디자인 시스템 ]
  카키(Khaki) 계열 색상 팔레트를 CSS 변수로 정의하고, Tailwind 레이어 컴포넌트
  (.btn-primary, .input-base 등)로 일관된 UI를 유지합니다.
  Pretendard 폰트(한국어 최적화)를 CDN으로 적용했습니다.

[ 6. 운동 일지 캘린더 뷰 ]
  react-calendar 라이브러리를 커스텀 스타일로 오버라이드하여
  날짜별 운동 기록을 시각적으로 확인할 수 있는 UI를 제공합니다.

[ 7. StrictMode 대응 ]
  React 18 StrictMode의 이중 실행(double invocation) 문제를 AuthProvider에서
  useRef로 초기화 플래그를 관리하여 인증 초기화가 중복 실행되지 않도록 처리합니다.
